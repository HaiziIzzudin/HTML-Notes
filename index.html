What is metadata?

  • An abstraction of the data.

  • Means that the relationship between data object and the metadata describing its functionally identical to the relationship between a book and its library catalog record.

What is Dublin Core Metadata Initiative (DCMI)? ...is a metadata element set intended to facilitate discovery of electronic resources.
Why does it favoured over other Metadata Schemas? Originally conceived for author - it has attracted the attention of formal resource description communities such as museums, libraries, government agencies and commercial organization.
Key characteristics of DCMI
  1. Simplicity
  2. Semantic interoperability
  3. International consensus
  4. Extensibility
  5. Modularity
Describe more about DCMI
  • Positioned as a simple information resources description format.
  • Aims to provide a basis for semantic interoperability between other, often more complicated formats.
  • DC benefits from active participation and promotion in some 20 countries in North America, Europe, Australia and Asia.
  • Representatives of the DC effort have helped to develop the Resources Description Framework (RDF), an infrastructure that supports coexistence of complementary, independently maintained metadata packages.
Give 7 properties used for DCMI schema.
  1. DC Title: The name or names by which a resource is formally known.
  2. DC Date: Description of any dates or ranges in the lifecycle of a resource and is typically associated with the creation or availability.
  3. DC Format: File format, the physical medium, or the dimension (the size or duration) of a resource. The information can be relevant to determine the equipment needed to display or operate a resource. To specify the different categories of format you should use extent and/or medium. To reference to the nature or genre of the content use type.
  4. DC Type: Description of the nature or genre of the content of a resource. To describe the physical or digital manifestation, use format.
  5. DC Creator: The person, an organization, or a service primarily responsible for making the content of a resource.
  6. DC Description: Description of the content of a resource. The description is a potentially rich source of indexable terms and assist the users in their selection of an appropriate resource.
  7. DC Subject: Relationship between a resource and another resource which is a topic of the first resource rsp. describes the intellectual content of the first resource.
4 Priciples of Metadata: 1. Modularity
  • Characterized by diverse sources of content, styles of content management, and approaches to resource description.
  • Allows designers of metadata schemas to create new assemblies based on established metadata schemas and benefit from observed best practice.
  • Data elements from different schemas, vocabularies and other building blocks can be combined in a syntactically and semantically interoperable way.
2. Extensibility
  • Must allow for extensions so that particular needs of a given application can be accommodated.
  • Some metadata elements are likely to be found in most metadata schemas, while others will be specific to particular applications or domains.
  • Must easily accommodate the notion of a base schema with additional elements that tailor a given application to local needs or domain-specific needs without compromising the interoperability provided by the base schema.
3. Refinement
  • metadata standards should allow schema designers to choose a level of detail appropriate to a given application.
  • 2 notions of refinement to consider:
    1. Addition of qualifiers that refine the meaning of an element. E.g.: Illustrator, author, composer, or sculptor are all particular types of general term creator.
    2. Specification of particular schemes or value sets that define the range of values for a given element. This way, semantic interoperability across applications can be increased by relying on a common value set.
4. Multilingualism
  • The Web affords unprecedented access to resources.
  • Complementary processes of internationalization and localization.
  • These two processes can sometimes work at cross-purposes. While global resource discovery is best served by internationalization, the needs of any given community may be better served by supporting local conventions.
9 Practicalities of Metadata:
  1. Application Profile
    Assemblage of metadata elements selected from one or more metadata schemas and combined in a compound schema.
  2. Syntax and Semantics
    Semantics is about meaning; syntax is about form.
  3. Association Models
    Embedded metadata resides within the markup of the resource. Associated metadata is maintained in files tightly coupled to the resources they describe. Third-Party metadata is maintained in a separate repository by an organization.
  4. Identifying and Naming Metadata Elements
    Particular tools and applications can use different presentation labels within their systems to make the labels more understandable and useful in a given linguistic, cultural, or domain context.
  5. Metadata Registries
    ...provide the means to identify and refer to established schemas and application profiles, potentially including the means for machine mapping among different schemas.
  6. Completeness of Description
    ...will be determined by policies and best practices designated by the agency creating the metadata, and those policies and practices will be guided by the functional requirements of services or applications.
  7. Mandatory Versus Optional Elements
    An element that is essential in one domain may not even be sensible in another, hence few, if any, elements in a general metadata set should be thought of as mandatory.
  8. Subjective and Objective Metadata
    Objective metadata can be machine generated in most instances, while Subjective metadata are specifically intended to represent a subjective evaluation.
  9. Automated Generation of Metadata
    ...that can be automated to some degree, and which can be expected to grow in importance as advances in such areas as natural language processing, data mining, profile and pattern recognition algorithms become more effective.

  1. Administrative metadata in managing and administering informatio resources.

What is Record Entity?
  • The Record Entity is a key component in the preservation of digital objects within a repository.
  • It is crucial for maintaining the digital provenance and authenticity of objects.
  • The Record Entity is associated with the preservation of intellectual entities and their link to digital objects.
  • It plays a vital role in aggregating information about actions affecting objects in the repository.
  • The accurate and trustworthy recording of events is essential for maintaining digital provenance and demonstrating the authenticity of objects.
  • Information recorded about events includes unique identifiers, event types, date and time of the event, detailed event descriptions, and coded outcomes.
  • The Record Entity is closely linked to the Intellectual Entity, which is defined as a conceptual unit for management and description, such as a book, map, photograph, or database, within the preservation repository.
What is Agent Entity?
  • The Agent Entity is a crucial component in the preservation of digital objects within a repository, responsible for recording additional information about agents, their roles in events, and their involvement in rights statements.
  • Agents are actors that have roles in events and rights statements, and can be people, organizations, or software applications.
  • The focus of the Data Dictionary is to identify the agent and classify the agent type (person, organization, or software), emphasizing the significance of unambiguous agent identification in metadata.
  • Agents influence objects only indirectly through events, and the role of the agent is a property of the Event entity, not of the Agent entity.
  • The Semantic Data Dictionary outlines mandatory and optional semantic units for agent identification and classification, with the mandatory semantic unit being the agentIdentifier.
  • Agent information serves to unambiguously identify an agent from all other entities, and it is crucial for maintaining digital provenance and demonstrating the authenticity of objects within the repository.
What is Function Entity?
  • The Function Entity in metadata plays a crucial role in preserving the entire metadata within a repository.
  • It contributes to the preservation of metadata by acting as an action or function.
  • The Function Entity encompasses various functional entities within theOpen Archival Information System (OAIS)model, including Ingest, Archival Storage, Data Management, Preservation Planning, and Access.
  • Each functional entity provides specific services and functions, such as accepting submission information packages, storing and managing archival information packages, populating and maintaining descriptive information, providing access to information products, and monitoring the environment to ensure long-term accessibility of stored information.
  • ThePreservation Planning Functional Entity, a part of the Function Entity, is responsible for monitoring the OAIS environment, providing recommendations and preservation plans to ensure the long-term accessibility and understandability of information stored in the OAIS.
Logical Schema of Metadata
  • The logical schema's data model is shaped by the physical component implementing it. In this perspective, a key concept is Schema, which, like a model's concepts, comprises Types of data model.
  • Transforming the logical schema into a physical one involves considering storage and performance aspects influenced by the Database Management System (DBMS) in use.
  • A logical schema portrays real-world objects, their properties, and relationships using familiar DBMS concepts like tables, attributes (columns), primary keys, and foreign keys.
Functional Schema of Metadata
  • Basic Functional Schema offers core schema structures for various manual and automated systems.
  • It resolves conflicts with configurational schemas.
  • Functions in this schema can be simple or complex based on property definitions.
  • In a database, there's a common schema for master tables, and Functional Schemas store additional information.
  • Functional Schema is flexible, allowing the creation of new entity types for high-level metadata representation.
Object Visualization of Metadata
  • Object visualization relies on image processing and neural networks.
  • It involves processing visual information about objects, including color and shape, as well as understanding spatial relations and performing transformations.
  • The web-based metadata and object visualization frontend presents a list of objects. Clicking an object triggers its visualization, revealing associated metadata.

Advantages of MODS/ Why use MODS?
  1. Not MARC: MODS is a derivative of MARC, but it is not MARC. It offers a more flexible and understandable alternative to MARCXML, allowing for richer descriptions with substructure and semantics that are a more natural fit in a library environment.
  2. Easier entry point: MODS is easier to use than MARC, as it does not require the use of codes and uses XML, which is a more widely understood and accessible format.
  3. More end-user friendly than MARCXML: MODS is simpler and more understandable than MARCXML, making it more accessible to end-users who may not be familiar with library-specific metadata standards.
  4. Useful middle ground: MODS offers a useful middle ground between MARC and simple Dublin Core, providing a balance between richness and flexibility in metadata representation.
  5. Richer than Dublin Core: MODS provides a richer metadata element set compared to Dublin Core, allowing for more detailed descriptions of digital objects.
  6. Hierarchical structure: MODS' hierarchical structure enables rich descriptions of complex digital objects, making it suitable for organizing and presenting diverse types of content.
  7. Compatibility with existing descriptions: The MODS element set is more compatible with existing descriptions than Dublin Core, making it easier to integrate with and build upon previous metadata standards.
  8. Compatibility with hierarchical METS objects: MODS is a rich descriptive element set that works well with hierarchical Metadata Encoding and Transmission Standard (METS) objects, enhancing its utility for digital libraries and archives.
  9. Flexibility and understandability: MODS offers a more flexible and understandable alternative to MARCXML, allowing for richer descriptions with substructure and semantics that are a more natural fit in a library environment.
Features of MODS
  1. Language-based tags: MODS uses language-based tags, which are more readily understandable and easier to use than numeric tags.
  2. Inherited semantics: MODS elements generally inherit semantics from the Machine-Readable Cataloging (MARC) standard, making it easier to understand and use for those familiar with MARC.
  3. Applicability to digital resources: MODS includes elements that are particularly applicable to digital resources, such as digital object identifiers (DOIs) and technical metadata.
  4. No specific rules for description: MODS does not assume the use of any specific rules for description, allowing for flexibility in how metadata is created and used.
  5. Reusable element descriptions: MODS element descriptions are reused throughout the schema, reducing redundancy and making it easier to create and maintain metadata.
  6. XML schema: MODS uses XML schema, which allows for flexibility and availability of freely available tools for creating, validating, and transforming metadata.

What is Shareable Metadata?
  1. Quality metadata that is accurate, consistent, and reliable.
  2. Promotes search interoperability, allowing for meaningful search results across diverse sets of metadata records.
  3. Human understandable outside of its local context, ensuring that the metadata can be comprehended by individuals not familiar with its original context.
  4. Useful outside of its local context, enabling the metadata to serve a purpose beyond its initial creation environment.
  5. Machine processable, allowing for automated processing and integration into different systems and workflows.
Implications/ Issues of Poor Metadata Practices
  1. Difficulty in Finding Records: Poor metadata practices can make it difficult to find records because they were not classified or indexed properly, or there is no thesaurus or other tools to facilitate access and retrieval.
  2. Inability to Relate Records: Poor metadata practices can make it difficult to relate records documenting a given work process to one another, making it impossible to understand the full 'story' of what occurred.
  3. Inability to Control Access and Use: Poor metadata practices can place the security of records at risk if metadata documenting access permissions, the security status of records, and the related security provisions are inadequate or missing.
  4. Risk to Authenticity and Integrity of Records: Poor metadata practices can bring into question the authenticity, validity, and integrity of certain records, making it difficult to authenticate records and control access and use.
  5. Business and Operational Issues: Poor metadata practices can make it difficult for staff to find and understand the records and information they need to achieve their operational goals, leading to business and operational issues.
  6. Legal and Regulatory Compliance Issues: Poor metadata practices can lead to legal and regulatory compliance issues, making it difficult for organizations to comply with laws and regulations.
  7. Lack of Trust in Government: Poor metadata practices can lead to a lack of trust in government's ability to manage its information, making it difficult to comply with the Freedom of Information law.
Example of Poor Metadata Practices
  1. Incomplete Supporting Documents: Support documents were incomplete, leading to overlooked information.
  2. Reuse of Payroll Numbers: Reusing a unique payroll number to add an employee to the payroll resulted in creating a ghost worker and defrauding the government.
  3. Lack of Linking Records: Records not linked by descriptions led to a flawed security policy that had to be immediately withdrawn and revised.
  4. Errors in Company Contact Details: Errors in the company's contact details in the vendor database led to sending a Purchase Order to the wrong address, causing customer frustration and lack of trust in the government.
How can we circumvent poorly managed metadata?
  1. Developing Metadata Standards: Developing metadata standards can help ensure that metadata is captured consistently and accurately.
  2. Implementing Records Management Policies: Implementing records management policies can help ensure that records are managed effectively throughout their lifecycle, including the capture and management of metadata.
  3. Providing Training and Education: Providing training and education to staff can help ensure that they understand the importance of metadata and how to manage it effectively.
  4. Using Technology: Using technology such as digital signatures, audit trails, and digital stamps can help authenticate records and control access and use.
  5. Retaining Records and Supporting Metadata: Retaining records and supporting metadata in a secure and accessible manner can help ensure that they are available when needed.
  6. Collaborating with Archivists: Collaborating with archivists can help ensure that metadata created in records-creating organizations, when combined with metadata provided by archivists, facilitate access and retrieval and support the management of the records as archival records through time.
